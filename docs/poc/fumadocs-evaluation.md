`public/content/**` (translations excluded) compile and render through

## TL;DR

Fumadocs is viable. Its build-time MDX compilation removes our worst class of bugs (Netlify ISR 404s caused by reading `public/content/*.md` at request time — see `next.config.js:255` where the directory is excluded from output tracing). With the integration hooks below, all 57 English tutorials render correctly through the Fumadocs route, with images, TOC, custom MdComponents, and our tutorials layout components.

The migration is not free. Each gotcha below is concrete and one-time:

1. **Loader rule collision.** `createMDX()` registers a global turbopack `*.{md,mdx}` rule that does **not** override our existing `*.md` raw-loader rule (specific patterns win over brace patterns). Files arrive as raw strings, React renders them as JSX with the whole file as the tag name (`Invalid tag: ---`). Fix: delete the `*.md` raw rule when Fumadocs is active.
2. **`*.yaml` blanket rule.** `createMDX()` adds a `*.yaml` turbopack rule for its meta collection that shadows our `yaml-loader` rule for `src/data/*.yaml`. Fix: re-apply our yaml rules after Fumadocs wraps the config. (Also: use `defineCollections({type:'doc'})` rather than `defineDocs` to skip the meta collection entirely, avoiding the equivalent webpack rule.)
3. **`mdxOptions` on a collection silences the preset.** Setting `collection.mdxOptions` bypasses Fumadocs' default plugin set (TOC extraction, image transform, code highlighting). Fix: call `applyMdxPreset({...})` explicitly and pass the resolver as `mdxOptions`.
4. **Preset's `remarkImage` clashes with our `rehypeImg`.** Default `useImport: true` rewrites `![](./x.png)` into ES module imports — our plugin expects raw `<img>` nodes with string `src`. Fix: `remarkImageOptions: false`.
5. **Preset's Shiki rejects unknown languages.** Tutorials use `yul`, `zokrates`, etc. that aren't in Shiki's default set. Existing pipeline highlights via `pre`/`code` component overrides, not at compile-time. Fix: `rehypeCodeOptions: false`.
6. **`format` is derived from file extension, not config.** `.md` files compile in MDX-as-Markdown mode regardless of `mdxOptions.format`. With `format: 'md'`, JSX components in content (`<Emoji>`, `<Alert>`, `<Card>` — used in 17/57 tutorials) are silently stripped. To render them, files must be renamed to `.mdx`, or `filePath` would need to be monkey-patched inside a plugin's `doc.vfile` hook. Out of scope for this PoC; documented as a real migration cost.
7. **i18n file layout mismatch.** Fumadocs expects sibling-suffix translations (`page.cn.md`). We use `public/content/translations/<locale>/<slug>/index.md`. Out of scope for this PoC; either restructure 8,052 files or write a custom Source adapter.

## What works in the PoC

- **All 57 tutorial pages render at 200**, with frontmatter title/description, body content, internal links, external links, and code blocks.
- **Images** render through our `rehypeImg` (adapted to Fumadocs in `src/lib/poc-fumadocs/rehypeImgForFumadocs.ts`) — width/height/aspectRatio/blur placeholder all populate; Next.js `<Image>` srcset is emitted as before.
- **Heading IDs** (our `{#kebab-id}` syntax) via `remark-heading-id`; anchor links work.
- **TOC** populated via Fumadocs' `remarkHeading` (within the preset) and surfaced as `page.data.toc` in the route.
- **Static params** generated by `tutorialsSource.generateParams()` — no hand-rolled `getPostSlugs` / `checkPathValidity`.
- **Custom MdComponents + tutorialsComponents** wire through `<MDXContent components={{...}} />`. The earlier "Functions cannot be passed to Client Components" error went away once the loader was actually compiling the MDX (the previous error was a side-effect of the file content being passed in as a string).

## What's stubbed / left out

- **Component-level JSX (`<Emoji>`, `<Alert>`, `<Card>`, `<EnvWarningBanner>`, …)** — silently stripped because `.md` extension forces MDX format='md'. 17/57 tutorials use one or more of these. Workaround in real migration: rename to `.mdx`, or patch Fumadocs' file-extension check.
- **Non-English locales** — `tutorialsSource.generateParams()` returns English-only paths; route hardcodes `locale: "en"`. Translations work only after the i18n adapter decision (see point 7).
- **Contributors / reading-time / lastEditLocaleTimestamp** — these come from filesystem + git in our current `data.ts`. With Fumadocs build-time compile they'd run during the source build step. Not wired in PoC.

## Why this is worth pursuing

`next.config.js:255` excludes `public/content` from `outputFileTracingExcludes`, so MD files **don't ship** in the Netlify serverless bundle. Combined with `unstable_cache` TTLs and runtime `readFile()` in `src/lib/md/import.ts:11`, this is the documented root cause of:

- `docs/solutions/integration-issues/netlify-isr-404-async-server-components.md`
- The recent `fix(contributors): use static cache` (commit `24d90e9d75`)
- The `fix(videos): clone frontmatter to prevent gray-matter cache pollution` (commit `3a4c55028a`)

Build-time MDX compilation (à la contentlayer) bundles every page as a JS module. The 404 class disappears by construction. Fumadocs' main draw for us is this property, not its UI components or search.

## What migrates cleanly

- **Routing.** `[...slug]/page.tsx` + `getPostSlugs` + `checkPathValidity` → one `source.generateParams()` call.
- **TOC.** `remarkInferToc` + `escapeHeadingIds` + `IdAnchor` → built into Fumadocs (`page.data.toc`).
- **Frontmatter typing.** `Frontmatter` in `src/lib/types.ts` → a Standard Schema (Zod) on the collection. (PoC skipped this to avoid pulling Zod 4 since we have Zod 3.)
- **Layout selection.** `getLayoutFromSlug` heuristic → one collection per layout (tutorials, docs, static, upgrade, …).
- **Heading IDs, images, custom MDX components** — all working in the PoC with one-line adapters.

## What doesn't migrate cleanly

| Current pattern | Fumadocs cost |
|---|---|
| `public/content/translations/<locale>/<slug>/index.md` | No native support. Restructure 8,052 files **or** write a custom `Source` adapter that buckets `.source` glob entries by locale (~150 LOC). |
| `.md` extension with embedded JSX components | Either rename ~17 tutorials (and ~all docs + static content that use `<Card>`, `<Alert>`, …) to `.mdx`, or patch the extension check in Fumadocs. |
| Crowdin sanitizer pipeline (`docs/solutions/logic-errors/*.md`) | Needs to run pre-compile (today it runs in `compile.ts`). With build-time compilation, this means sanitizing at content-source time — aligns better with our intl-pipeline architecture, but still a port. |
| `remarkPreserveJsx` (MDX v1 unwrapping) | Not needed for the PoC content; may surface once `.mdx` rename happens and v1 patterns are re-parsed as JSX. |
| Reading-time / contributors / last-edit timestamp | Today: filesystem + git in `data.ts`. With Fumadocs: a plugin's `doc.frontmatter` hook or a separate prepass. Same effort, different timing. |
| Plaiceholder blur on hero images | Today in `compile.ts:64`. Wire as a small remark/rehype plugin. Same effort. |

## Architectural questions to answer before committing

1. **Build perf at scale.** ~~PoC compiles 57 files; the full 8,052 is two orders of magnitude larger~~ — **answered**. The PoC was expanded to compile all 326 English pages; cold build cost is +12 s of MDX compile (~37 ms/page) and +10 s wall time (+10 %) on a 1 m 42 s baseline. See `PROGRESS.md` *Full-scope expansion* for the table. Per-page extrapolation, not raw file count, is the right model — and at that rate even the full locale matrix is tens of minutes at worst, not hours. Build perf is not the blocker.
2. **SSG vs ISR.** Today: SSG for EN, ISR for other locales. With Fumadocs, full SSG becomes natural since content is bundled — eliminates the ISR-404 class entirely. Need to confirm Netlify build-minutes budget tolerates the shift.
3. **i18n adapter.** Decision needed: restructure content directories (large one-time intl-pipeline change) or write a custom source (~150 LOC, non-standard layer). Restructure is cleaner long-term. **This is now the only remaining go/no-go gate.**
4. **`.md` → `.mdx` rename.** Touch every content file. Coordinate with intl-pipeline + Crowdin source format expectations. The full-scope PoC confirmed this is necessary for two independent reasons: (a) ~18 % of EN files have embedded JSX components that silently strip in `.md` mode, (b) ~2 % have inline HTML in headings that breaks Fumadocs' `rehype-toc` (worked around in the PoC by a `rehypeStripRaw` adapter; the rename removes that need).

## Recommendation

Worth pursuing on a separate branch, in this order:

1. ~~Rename to `.mdx`~~ Still recommended as first step (one PR, mechanical) — full-scope PoC measured that 18 % of EN files use JSX components that strip in `.md` mode and 2 % have inline HTML that breaks rehype-toc without a workaround.
2. ~~Build full prod compile with all 326 EN pages in Fumadocs collections. Compare `pnpm build` time and bundle size to today.~~ **Done** — see `PROGRESS.md` *Full-scope expansion*. +10 s wall time, +12 s compile, +300 MB `.next` (most of which goes away when the existing pipeline is removed). **Go on perf.**
3. Design the i18n adapter (or restructure content dirs). **Go/no-go on translations.** This is the only remaining gate.
4. Migrate one layout end-to-end (tutorials first — 57 × 25 files).

Don't migrate piecemeal across layouts. Two MDX compilers, two TOC implementations, and two component maps running side-by-side is worse than either system alone.

## How to reproduce / explore the PoC

```bash
pnpm install        # fumadocs-mdx, fumadocs-core already in devDeps
npx fumadocs-mdx    # regenerates .source/
pnpm dev
# Visit:
#   http://localhost:3000/en/poc-fumadocs/                     -> index of 57 tutorials
#   http://localhost:3000/en/poc-fumadocs/<any-slug>/          -> any tutorial, rendered
#   http://localhost:3000/en/developers/tutorials/...          -> existing route, unaffected
```

To remove the PoC: revert `next.config.js`, delete `source.config.ts`, `.source/`, `src/lib/poc-fumadocs/`, `app/[locale]/poc-fumadocs/`, run `pnpm remove fumadocs-mdx fumadocs-core`.

## Notes on the integration patches required

The "extra plumbing" in `source.config.ts` and `next.config.js` looks small but tells you something:

- Fumadocs' defaults aim at a greenfield docs site (rename your files to `.mdx`, use their Shiki, use their `useImport: true` image pipeline). The closer your project gets to those defaults, the fewer patches.
- Where ethereum.org has its own opinion (image preplaceholder, Prism-based syntax highlighting, raw markdown extension, custom yaml-loader for `src/data`), Fumadocs is "configurable but not gracefully" — you turn defaults off rather than slot in.

If migration goes ahead, expect the PR description to list a similar set of "things we disabled in Fumadocs to keep our existing behavior." That is normal for a content-pipeline swap; not normal for "a simple drop-in."
