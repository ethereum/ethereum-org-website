Web2は、今日私たちが知っているインターネットのバージョンを指します。個人データと引き換えにサービスを提供する企業によって支配されているインターネットです。[イーサリアム](/)の文脈におけるWeb3は、ブロックチェーン上で実行される分散型アプリケーション (dapp) を指します。これらは、個人データを収益化することなく、誰でも参加できるアプリです。

より初心者向けのリソースをお探しですか？[Web3の紹介](/web3/)をご覧ください。

## Web3の利点 {#web3-benefits}

多くのWeb3開発者は、イーサリアムに固有の分散化を理由に、分散型アプリケーション (dapp) を構築することを選択しています。

- ネットワーク上の誰でもサービスを使用する許可を持っています。言い換えれば、許可は必要ありません。
- 誰もあなたをブロックしたり、サービスへのアクセスを拒否したりすることはできません。
- 支払いは、ネイティブトークンであるイーサ (ETH) を通じて組み込まれています。
- イーサリアムはチューリング完全であり、ほぼ何でもプログラムできることを意味します。

## 実用的な比較 {#practical-comparisons}

| Web2                                                                       | Web3                                                                                                           |
| -------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| ツイッターは、任意のアカウントやツイートを検閲できる                                    | Web3のツイートは、制御が分散型であるため検閲できない                                             |
| 決済サービスは、特定の種類の仕事に対する支払いを許可しない決定を下すことができる | Web3の決済アプリは個人データを必要とせず、支払いを妨げることができない                                          |
| ギグエコノミーアプリのサーバーがダウンし、労働者の収入に影響を与える可能性がある        | Web3のサーバーはダウンしない。バックエンドとして、数千台のコンピューターからなる分散型ネットワークであるイーサリアムを使用しているためである |

これは、すべてのサービスを分散型アプリケーション (dapp) に変える必要があるという意味ではありません。これらの例は、Web2サービスとWeb3サービスの主な違いを説明するためのものです。

## Web3の制限 {#web3-limitations}

現在のところ、Web3にはいくつかの制限があります。

- スケーラビリティ – Web3では分散型であるため、トランザクションの処理が遅くなります。支払いなどの状態の変更は、ノードによって処理され、ネットワーク全体に伝播される必要があります。
- UX – Web3アプリケーションとのやり取りには、追加の手順、ソフトウェア、および学習が必要になる場合があります。これが普及のハードルになる可能性があります。
- アクセシビリティ – 最新のウェブブラウザへの統合が不足しているため、ほとんどのユーザーにとってWeb3はアクセスしにくいものになっています。
- コスト – ブロックチェーンへのコードの配置は高価であるため、成功している分散型アプリケーション (dapp) のほとんどは、コードのごく一部のみをブロックチェーン上に配置しています。

## 中央集権化と分散化の比較 {#centralization-vs-decentralization}

以下の表では、中央集権型および分散型のデジタルネットワークの、大まかな利点と欠点をいくつか挙げています。

| 中央集権型システム                                                                                                                                                                                                    | 分散型システム                                                                                                                                                                                                            |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ネットワークの直径が小さい (すべての参加者が中央機関に接続されている)。豊富な計算リソースを持つ中央機関が伝播を処理するため、情報が迅速に伝播する。                   | ネットワーク上で最も離れた参加者同士は、多くのエッジを隔てている可能性がある。ネットワークの一方の端からブロードキャストされた情報が、もう一方の端に到達するまでに長い時間がかかる場合がある。                                    |
| 通常、パフォーマンスが高く (スループットが高く、消費される総計算リソースが少ない)、実装が容易である。                                                                                                  | 通常、パフォーマンスが低く (スループットが低く、消費される総計算リソースが多い)、実装がより複雑である。                                                                                                         |
| データが競合した場合の解決が明確で容易である。究極の信頼できる情報源は中央機関である。                                                                                                 | 参加者が同期すべきデータの状態について、ピアが相反する主張をした場合、紛争解決のための (しばしば複雑な) プロトコルが必要になる。                                                        |
| 単一障害点がある。悪意のあるアクターが中央機関を標的にすることで、ネットワークをダウンさせることができる可能性がある。                                                                                                     | 単一障害点がない。参加者の大部分が攻撃されたり排除されたりしても、ネットワークは機能し続けることができる。                                                                                                        |
| ネットワーク参加者間の調整がはるかに容易であり、中央機関によって処理される。中央機関は、ネットワーク参加者にアップグレードやプロトコルの更新などを、ほとんど摩擦なく強制することができる。 | ネットワークレベルの決定やプロトコルのアップグレードなどにおいて、最終的な決定権を持つ単一のエージェントが存在しないため、調整が困難な場合が多い。最悪の場合、プロトコルの変更について意見の不一致があると、ネットワークが分裂しやすくなる。 |
| 中央機関はデータを検閲し、ネットワークの一部がネットワークの他の部分とやり取りするのを遮断する可能性がある。                                                                                         | 情報がネットワーク全体に伝播する経路が多数あるため、検閲ははるかに困難である。                                                                                                                                         |
| ネットワークへの参加は中央機関によって制御される。                                                                                                                                                   | 誰でもネットワークに参加できる。「ゲートキーパー」は存在しない。理想的には、参加コストは非常に低い。                                                                                                               |

これらは一般的な傾向であり、すべてのネットワークに当てはまるとは限らないことに注意してください。さらに、現実にはネットワークが中央集権型か分散型かの度合いはスペクトラム上にあり、完全に中央集権型、あるいは完全に分散型のネットワークは存在しません。
