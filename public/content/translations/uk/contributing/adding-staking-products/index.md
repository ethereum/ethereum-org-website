Ми хочемо переконатися, що пропонуємо найкращі можливі ресурси, зберігаючи при цьому безпеку та впевненість користувачів.

Будь-хто може запропонувати додати продукт або послугу для стейкінгу на ethereum.org. Якщо ми щось пропустили, **[будь ласка, запропонуйте це](https://github.com/ethereum/ethereum-org-website/issues/new?assignees=&labels=feature+%3Asparkles%3A%2Ccontent+%3Afountain_pen%3A&template=suggest_staking_product.yaml)!**

Наразі ми розміщуємо продукти та послуги для стейкінгу на таких сторінках:

- [Соло-стейкінг](/staking/solo/)
- [Стейкінг як послуга](/staking/saas/)
- [Стейкінг-пули](/staking/pools/)

Доказ частки (PoS) у сигнальному ланцюзі працює з 1 грудня 2020 року. Хоча стейкінг все ще є відносно новим, ми спробували створити справедливу та прозору систему для розгляду на ethereum.org, але критерії розміщення будуть змінюватися та розвиватися з часом, і в кінцевому підсумку залишаються на розсуд команди вебсайту ethereum.org.

## Система прийняття рішень {#the-decision-framework}

Рішення про розміщення продукту на ethereum.org не залежить від якогось одного фактора. Під час прийняття рішення про розміщення продукту або послуги розглядається кілька критеріїв разом. Чим більше цих критеріїв виконано, тим більша ймовірність того, що продукт буде розміщено.

**По-перше, до якої категорії належить продукт або послуга?**

- Інструменти для вузлів або клієнтів
- Управління ключами
- Стейкінг як послуга (SaaS)
- Стейкінг-пул

Наразі ми розміщуємо лише продукти або послуги в цих категоріях.

### Критерії для включення {#criteria-for-inclusion}

Заявки на продукти або послуги для стейкінгу оцінюватимуться за такими критеріями:

**Коли було запущено проєкт або послугу?**

- Чи є докази того, коли продукт або послуга стали доступними для громадськості?
- Це використовується для визначення оцінки «перевіреності на практиці» продукту.

**Чи активно підтримується проєкт?**

- Чи є активна команда, яка розробляє проєкт? Хто бере участь?
- Розглядатимуться лише ті продукти, які активно підтримуються.

**Чи продукт або послуга не мають довірених/людських посередників?**

- Які кроки на шляху користувача вимагають довіри до людей щодо зберігання ключів від їхніх коштів або належного розподілу винагород?
- Це використовується для визначення оцінки «бездовірчості» продукту або послуги.

**Чи надає проєкт точну та надійну інформацію?**

- Вкрай важливо, щоб на вебсайті продукту була актуальна, точна інформація, яка не вводить в оману, особливо якщо вона стосується протоколу Етеріум або інших пов'язаних технологій.
- Заявки, що містять дезінформацію, застарілі дані або потенційно оманливі заяви про Етеріум чи інші відповідні теми, не будуть розміщені або будуть видалені, якщо вони вже розміщені.

**Які платформи підтримуються?**

- тобто Linux, macOS, Windows, iOS, Android

#### Програмне забезпечення та смарт-контракти {#software-and-smart-contracts}

Для будь-якого спеціального програмного забезпечення або смарт-контрактів:

**Чи все має відкритий вихідний код?**

- Проєкти з відкритим вихідним кодом повинні мати загальнодоступний репозиторій вихідного коду.
- Це використовується для визначення оцінки «відкритого вихідного коду» продукту.

**Чи вийшов продукт зі стадії _бета-розробки_?**

- На якому етапі циклу розробки знаходиться продукт?
- Продукти на стадії бета-тестування не розглядаються для включення на ethereum.org.

**Чи пройшло програмне забезпечення зовнішній аудит безпеки?**

- Якщо ні, чи є плани щодо проведення зовнішнього аудиту?
- Це використовується для визначення оцінки «аудиту» продукту.

**Чи є у проєкті програма винагород за знайдені помилки (bug bounty)?**

- Якщо ні, чи є плани щодо створення програми винагород за знайдені помилки безпеки?
- Це використовується для визначення оцінки «bug bounty» продукту.

#### Інструменти для вузлів або клієнтів {#node-or-client-tooling}

Для програмних продуктів, пов'язаних із налаштуванням, управлінням або міграцією вузлів чи клієнтів:

**Які клієнти рівня консенсусу (тобто Лайтхаус, Теку, Німбус, Призм, Grandine) підтримуються?**

- Які клієнти підтримуються? Чи може користувач вибирати?
- Це використовується для визначення оцінки «мультиклієнтності» продукту.

#### Стейкінг як послуга {#staking-as-a-service}

Для [списків стейкінгу як послуги](/staking/saas/) (тобто делегованого управління вузлом):

**Які комісії пов'язані з використанням послуги?**

- Яка структура комісій, наприклад, чи є щомісячна плата за послугу?
- Чи є додаткові вимоги до стейкінгу?

**Чи вимагається від користувачів реєстрація акаунта?**

- Чи може хтось використовувати послугу без дозволу або KYC?
- Це використовується для визначення оцінки «бездозвільності» продукту.

**Хто зберігає ключі підписання та ключі виведення?**

- До яких ключів користувач зберігає доступ? До яких ключів отримує доступ послуга?
- Це використовується для визначення оцінки «бездовірчості» продукту.

**Яка різноманітність клієнтів керованих вузлів?**

- Який відсоток ключів валідаторів працює на клієнті рівня консенсусу (CL), що становить більшість?
- Станом на останнє редагування, Призм є клієнтом рівня консенсусу, який використовується більшістю операторів вузлів, що є небезпечним для мережі. Якщо будь-який клієнт CL наразі використовується понад 33% мережі, ми запитуємо дані, пов'язані з його використанням.
- Це використовується для визначення оцінки «різноманітності клієнтів» продукту.

#### Стейкінг-пул {#staking-pool}

Для [послуг спільного стейкінгу](/staking/pools/):

**Який мінімум ETH вимагається для стейкінгу?**

- наприклад, 0.01 ETH

**Які комісії або вимоги до стейкінгу передбачені?**

- Який відсоток винагород стягується як комісія?
- Чи є додаткові вимоги до стейкінгу?

**Чи є токен ліквідності?**

- Які токени залучені? Як вони працюють? Які адреси контрактів?
- Це використовується для визначення оцінки «токена ліквідності» продукту.

**Чи можуть користувачі брати участь як оператори вузла?**

- Що потрібно для запуску клієнтів валідаторів з використанням спільних коштів?
- Чи вимагає це дозволу від особи, компанії або DAO?
- Це використовується для визначення оцінки «бездозвільних вузлів» продукту.

**Яка різноманітність клієнтів операторів вузлів пулу?**

- Який відсоток операторів вузлів використовує клієнт рівня консенсусу (CL), що становить більшість?
- Станом на останнє редагування, Призм є клієнтом рівня консенсусу, який використовується більшістю операторів вузлів, що є небезпечним для мережі. Якщо будь-який клієнт CL наразі використовується понад 33% мережі, ми запитуємо дані, пов'язані з його використанням.
- Це використовується для визначення оцінки «різноманітності клієнтів» продукту.

### Інші критерії: бажані переваги {#other-criteria}

**Які інтерфейси користувача підтримуються?**

- тобто браузерний застосунок, десктопний застосунок, мобільний застосунок, CLI

**Для інструментів вузла, чи надає програмне забезпечення простий спосіб перемикання між клієнтами?**

- Чи може користувач легко та безпечно змінювати клієнтів за допомогою інструмента?

**Для SaaS, скількома валідаторами наразі керує послуга?**

- Це дає нам уявлення про поточне охоплення вашої послуги.

## Як ми відображаємо результати {#product-ordering}

Наведені вище [критерії для включення](#criteria-for-inclusion) використовуються для розрахунку сукупної оцінки для кожного продукту або послуги. Це використовується як засіб сортування та демонстрації продуктів, які відповідають певним об'єктивним критеріям. Чим більше критеріїв підтверджено доказами, тим вище буде відсортовано продукт, а при однаковій кількості балів порядок буде випадковим під час завантаження.

Логіка коду та ваги для цих критеріїв наразі містяться в [цьому компоненті JavaScript](https://github.com/ethereum/ethereum-org-website/blob/dev/src/components/Staking/StakingProductsCardGrid/index.tsx#L350) у нашому репозиторії.
