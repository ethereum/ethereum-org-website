# Додавання продуктів або послуг для стейкінгу {#adding-staking-products-or-services}

Ми хочемо бути впевненими, що зібрали найкращі ресурси, забезпечуючи при цьому захист і довіру користувачів.

Будь-хто може внести пропозицію про додавання продуктів або послуг для стейкінгу на ethereum.org. Якщо ми щось пропустили, **[будь ласка, запропонуйте це](https://github.com/ethereum/ethereum-org-website/issues/new?assignees=&labels=feature+%3Asparkles%3A%2Ccontent+%3Afountain_pen%3A&template=suggest_staking_product.yaml)!**

На цих сторінках ви знайдете перелік продуктів та послуг для стейкінгу:

- [Соло-стейкінг](/staking/solo/)
- [Стейкінг як послуга](/staking/saas/)
- [Пули для стейкінгу](/staking/pools/)

Доказ частки в Beacon Chain працює з 1 грудня 2020 року. Хоча стейкінг все ще є відносно новим явищем, ми спробували створити справедливі і зрозумілі рамки для його розгляду на ethereum.org, але критерії лістингу будуть змінюватися і розвиватися з часом, і в кінцевому підсумку залишаються на розсуд команди вебсайту ethereum.org.

## Структура прийняття рішень {#the-decision-framework}

Рішення про додавання продукту на ethereum.org не залежить від якогось одного фактору. При прийнятті рішення про додавання товару чи послуги до списку враховуються декілька критеріїв. Чим більше з цих критеріїв дотримано, тим більша ймовірність того, що компанія потрапить до списку.

**По-перше, до якої категорії належить продукт або послуга?**

- Інструментарій для вузла або клієнта
- Керування ключами
- Стейкінг як послуга (SaaS)
- Пул для стейкінгу

Наразі ми розміщуємо продукти або послуги лише в цих категоріях.

### Критерії для включення {#criteria-for-inclusion}

Заявки на додавання продуктів або послуг для стейкінгу оцінюватимуться за такими критеріями:

**Коли було запущено проєкт або послугу?**

- Чи є докази того, коли продукт або послуга стали доступними для громадськості?
- Це використовується для визначення оцінки продукту "перевірений у боях".

**Чи проєкт активно підтримується?**

- Чи є активна команда, що розробляє проєкт? Хто бере участь?
- Розглядатимуться лише продукти, які активно підтримуються.

**Чи вільний продукт або послуга від довірених/людських посередників?**

- Які етапи шляху користувача вимагають довіри до людей, які або тримають ключі від їхніх коштів, або належним чином розподіляють винагороди?
- Це використовується для визначення оцінки продукту або послуги "без довіри".

**Чи надає проєкт точну та надійну інформацію?**

- Важливо, щоб вебсайт продукту містив актуальну, точну та неоманливу інформацію, особливо якщо вона стосується протоколу Ethereum або інших пов'язаних технологій.
- Заявки, що містять дезінформацію, застарілі дані або потенційно оманливі твердження про Ethereum чи інші відповідні теми, не будуть додані до списку або будуть видалені, якщо вони вже в ньому є.

**Які платформи підтримуються?**

- тобто Linux, macOS, Windows, iOS, Android

#### Програмне забезпечення та смарт-контракти {#software-and-smart-contracts}

Для будь-якого залученого спеціального програмного забезпечення або смарт-контрактів:

**Чи все має відкритий код?**

- Проєкти з відкритим кодом повинні мати загальнодоступне сховище вихідного коду
- Це використовується для визначення оцінки продукту "відкритий код".

**Чи продукт вийшов із _бета_-розробки?**

- На якому етапі циклу розробки знаходиться продукт?
- Продукти на стадії бета-тестування не розглядаються для включення на ethereum.org

**Чи пройшло програмне забезпечення зовнішній аудит безпеки?**

- Якщо ні, чи є плани провести зовнішній аудит?
- Це використовується для визначення оцінки продукту "проаудований".

**Чи є в проєкті програма винагород за виявлення помилок (bug bounty)?**

- Якщо ні, чи є плани створити програму винагород за виявлення помилок безпеки?
- Це використовується для визначення оцінки продукту "bug bounty".

#### Інструментарій для вузла або клієнта {#node-or-client-tooling}

Для програмних продуктів, пов’язаних із налаштуванням, керуванням або міграцією вузла чи клієнта:

**Які клієнти рівня консенсусу (тобто Lighthouse, Teku, Nimbus, Prysm, Grandine) підтримуються?**

- Які клієнти підтримуються? Чи може користувач вибирати?
- Це використовується для визначення оцінки продукту "багатоклієнтський".

#### Стейкінг як послуга {#staking-as-a-service}

Для списків [стейкінгу як послуги](/staking/saas/) (тобто делегованої роботи вузла):

**Які комісії пов'язані з використанням послуги?**

- Яка структура комісій, наприклад, чи є щомісячна плата за послугу?
- Чи є якісь додаткові вимоги до стейкінгу?

**Чи повинні користувачі реєструвати обліковий запис?**

- Чи може хтось користуватися послугою без дозволу або KYC?
- Це використовується для визначення оцінки продукту "бездозвольний".

**Хто володіє ключами підпису та ключами для виведення коштів?**

- До яких ключів користувач зберігає доступ? До яких ключів сервіс отримує доступ?
- Це використовується для визначення оцінки продукту "без довіри".

**Яке розмаїття клієнтів у вузлів, що експлуатуються?**

- Який відсоток ключів валідатора виконується клієнтом більшості на рівні консенсусу (CL)?
- На момент останнього редагування Prysm є клієнтом рівня консенсусу, який використовується більшістю операторів вузлів, що небезпечно для мережі. Якщо будь-який клієнт CL наразі використовується понад 33% мережі, ми просимо надати дані щодо його використання.
- Це використовується для визначення оцінки продукту "різноманітні клієнти".

#### Пул для стейкінгу {#staking-pool}

Для [сервісів об’єднаного стейкінгу](/staking/pools/):

**Яка мінімальна кількість ETH потрібна для стейкінгу?**

- напр., 0,01 ETH

**Які комісії або вимоги до стейкінгу?**

- Який відсоток винагород стягується як комісія?
- Чи є якісь додаткові вимоги до стейкінгу?

**Чи є токен ліквідності?**

- Які токени залучені? Як вони працюють? Які адреси контрактів?
- Це використовується для визначення оцінки продукту "токен ліквідності".

**Чи можуть користувачі брати участь як оператори вузлів?**

- Що потрібно для запуску клієнтів-валідаторів із використанням об'єднаних коштів?
- Чи потрібен для цього дозвіл від окремої особи, компанії чи DAO?
- Це використовується для визначення оцінки продукту "бездозвольні вузли".

**Яке розмаїття клієнтів серед операторів вузлів пулу?**

- Який відсоток операторів вузлів використовує клієнт більшості на рівні консенсусу (CL)?
- На момент останнього редагування Prysm є клієнтом рівня консенсусу, який використовується більшістю операторів вузлів, що небезпечно для мережі. Якщо будь-який клієнт CL наразі використовується понад 33% мережі, ми просимо надати дані щодо його використання.
- Це використовується для визначення оцінки продукту "різноманітні клієнти".

### Інші критерії: бажані, але не обов'язкові {#other-criteria}

**Які користувацькі інтерфейси підтримуються?**

- тобто браузерний застосунок, настільний застосунок, мобільний застосунок, CLI

**Щодо інструментарію для вузлів: чи забезпечує програмне забезпечення простий спосіб перемикання між клієнтами?**

- Чи може користувач легко та безпечно змінювати клієнтів за допомогою інструмента?

**Для SaaS: скільки валідаторів наразі обслуговується сервісом?**

- Це дає нам уявлення про охоплення вашої послуги на цей час.

## Як ми відображаємо результати {#product-ordering}

Наведені вище [критерії для включення](#criteria-for-inclusion) використовуються для розрахунку сукупного балу для кожного продукту або послуги. Це використовується як засіб для сортування та демонстрації продуктів, які відповідають певним об'єктивним критеріям. Чим більше критеріїв підтверджено доказами, тим вище буде відсортовано продукт, а у разі однакової кількості балів порядок буде рандомізовано під час завантаження.

Логіка коду та ваги для цих критеріїв наразі містяться в [цьому компоненті JavaScript](https://github.com/ethereum/ethereum-org-website/blob/dev/src/components/Staking/StakingProductsCardGrid.js#L350) у нашому репозиторії.
