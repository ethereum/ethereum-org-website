इथेरियम ब्लॉकचेनला विनंत्या करण्यासाठी WebSockets आणि Alchemy वापरण्याबाबतचे हे एक प्राथमिक स्तरावरील मार्गदर्शक आहे.

## WebSockets विरुद्ध HTTP {#websockets-vs-http}

HTTP च्या विपरीत, WebSockets सह, जेव्हा तुम्हाला विशिष्ट माहिती हवी असते तेव्हा तुम्हाला सतत विनंत्या करण्याची आवश्यकता नसते. WebSockets तुमच्यासाठी नेटवर्क कनेक्शन राखतात (जर योग्यरित्या केले असेल) आणि बदलांसाठी ऐकतात.

कोणत्याही नेटवर्क कनेक्शनप्रमाणे, तुम्ही असे गृहीत धरू नये की WebSocket कोणत्याही व्यत्ययाशिवाय कायमचे खुले राहील, परंतु ड्रॉप झालेले कनेक्शन आणि मॅन्युअली पुन्हा कनेक्ट करणे योग्यरित्या हाताळणे आव्हानात्मक असू शकते. WebSockets चा आणखी एक तोटा असा आहे की तुम्हाला प्रतिसादात HTTP स्टेटस कोड मिळत नाहीत, तर फक्त त्रुटी संदेश मिळतो.

​[Alchemy Web3](https://docs.alchemy.com/reference/api-overview) कोणत्याही कॉन्फिगरेशनची आवश्यकता नसताना WebSocket मधील बिघाड आणि पुनर्प्रयत्नांसाठी स्वयंचलितपणे हाताळणी जोडते.

## हे वापरून पहा {#try-it-out}

WebSockets ची चाचणी करण्याचा सर्वात सोपा मार्ग म्हणजे WebSocket विनंत्या करण्यासाठी [wscat](https://github.com/websockets/wscat) सारखे कमांड लाइन टूल इन्स्टॉल करणे. wscat वापरून, तुम्ही खालीलप्रमाणे विनंत्या पाठवू शकता:

_टीप: जर तुमच्याकडे Alchemy खाते असेल तर तुम्ही `demo` च्या जागी तुमची स्वतःची API की वापरू शकता. [मोफत Alchemy खात्यासाठी येथे साइन अप करा!](https://auth.alchemy.com/signup)_

```
wscat -c wss://eth-mainnet.ws.alchemyapi.io/ws/demo

>  {"jsonrpc":  "2.0", "id": 0, "method":  "eth_gasPrice"}

<  {"jsonrpc":  "2.0", "result":  "0xb2d05e00", "id": 0}
```

## WebSockets कसे वापरावे {#how-to-use-websockets}

सुरुवात करण्यासाठी, तुमच्या ॲपसाठी WebSocket URL वापरून WebSocket उघडा. तुम्ही [तुमच्या डॅशबोर्डमध्ये](https://dashboard.alchemy.com/) ॲपचे पेज उघडून आणि "View Key" वर क्लिक करून तुमच्या ॲपची WebSocket URL शोधू शकता. लक्षात घ्या की WebSockets साठी तुमच्या ॲपची URL ही HTTP विनंत्यांसाठी असलेल्या URL पेक्षा वेगळी आहे, परंतु दोन्ही "View Key" वर क्लिक करून शोधल्या जाऊ शकतात.

![Where to find your WebSocket URL in your Alchemy dashboard](./use-websockets.gif)

[Alchemy API संदर्भामध्ये](https://www.alchemy.com/docs/reference/api-overview) सूचीबद्ध केलेल्या कोणत्याही API चा वापर WebSocket द्वारे केला जाऊ शकतो. असे करण्यासाठी, HTTP POST विनंतीच्या मुख्य भागामध्ये पाठवला जाणारा पेलोड वापरा, परंतु त्याऐवजी तो पेलोड WebSocket द्वारे पाठवा.

## Web3 सह {#with-web3}

Web3 सारखी क्लायंट लायब्ररी वापरताना WebSockets वर जाणे सोपे आहे. तुमचा Web3 क्लायंट इन्स्टॅन्शिएट करताना HTTP ऐवजी फक्त WebSocket URL पास करा. उदाहरणार्थ:

```js
const web3 = new Web3("wss://eth-mainnet.ws.alchemyapi.io/ws/your-api-key")

web3.eth.getBlockNumber().then(console.log) // -> 7946893
```

## सबस्क्रिप्शन API {#subscription-api}

WebSocket द्वारे कनेक्ट केलेले असताना, तुम्ही दोन अतिरिक्त पद्धती वापरू शकता: `eth_subscribe` आणि `eth_unsubscribe`. या पद्धती तुम्हाला विशिष्ट घटना ऐकण्याची आणि त्वरित सूचित करण्याची अनुमती देतील.

### `eth_subscribe` {#eth-subscribe}

निर्दिष्ट केलेल्या घटनांसाठी नवीन सबस्क्रिप्शन तयार करते. [`eth_subscribe` बद्दल अधिक जाणून घ्या](https://docs.alchemy.com/reference/eth-subscribe).

#### पॅरामीटर्स {#parameters}

1. सबस्क्रिप्शनचे प्रकार
2. पर्यायी पॅरामीटर्स

पहिला आर्ग्युमेंट कोणत्या प्रकारच्या घटनेसाठी ऐकायचे ते निर्दिष्ट करतो. दुसऱ्या आर्ग्युमेंटमध्ये अतिरिक्त पर्याय असतात जे पहिल्या आर्ग्युमेंटवर अवलंबून असतात. विविध वर्णन प्रकार, त्यांचे पर्याय आणि त्यांचे घटना पेलोड खाली वर्णन केले आहेत.

#### रिटर्न्स {#returns}

सबस्क्रिप्शन ID: हा ID प्राप्त झालेल्या कोणत्याही घटनांना जोडला जाईल, आणि `eth_unsubscribe` वापरून सबस्क्रिप्शन रद्द करण्यासाठी देखील वापरला जाऊ शकतो.

#### सबस्क्रिप्शन घटना {#subscription-events}

सबस्क्रिप्शन सक्रिय असताना, तुम्हाला घटना प्राप्त होतील जे खालील फील्ड्स असलेले ऑब्जेक्ट्स आहेत:

- `jsonrpc`: नेहमी "2.0"
- `method`: नेहमी "eth_subscription"
- `params`: खालील फील्ड्स असलेला एक ऑब्जेक्ट:
  - `subscription`: हे सबस्क्रिप्शन तयार करणाऱ्या `eth_subscribe` कॉलद्वारे परत केलेला सबस्क्रिप्शन ID.
  - `result`: एक ऑब्जेक्ट ज्याची सामग्री सबस्क्रिप्शनच्या प्रकारानुसार बदलते.

#### सबस्क्रिप्शनचे प्रकार {#subscription-types}

3. `logs`

निर्दिष्ट फिल्टर निकषांशी जुळणाऱ्या नव्याने जोडलेल्या ब्लॉक्सचा भाग असलेले लॉग्स उत्सर्जित करते.

जेव्हा चेन रि-ऑर्ग होते, तेव्हा जुन्या चेनवरील ब्लॉक्सचा भाग असलेले लॉग्स `removed` प्रॉपर्टी `true` वर सेट करून पुन्हा उत्सर्जित केले जातील. पुढे, नवीन चेनवरील ब्लॉक्सचा भाग असलेले लॉग्स उत्सर्जित केले जातात, याचा अर्थ असा की रि-ऑर्गच्या बाबतीत एकाच व्यवहारासाठी अनेक वेळा लॉग्स पाहणे शक्य आहे.

पॅरामीटर्स

1. खालील फील्ड्स असलेला एक ऑब्जेक्ट:
   - `address` (पर्यायी): एकतर पत्ता दर्शवणारी स्ट्रिंग किंवा अशा स्ट्रिंगचा ॲरे.
     - फक्त यापैकी एका पत्त्यावरून तयार केलेले लॉग्स उत्सर्जित केले जातील.
   - `topics`: टॉपिक स्पेसिफायर्सचा ॲरे.
     - प्रत्येक टॉपिक स्पेसिफायर एकतर `null`, टॉपिक दर्शवणारी स्ट्रिंग किंवा स्ट्रिंगचा ॲरे असतो.
     - ॲरेमधील प्रत्येक स्थान जे `null` नाही ते उत्सर्जित लॉग्सना फक्त त्या स्थानावर दिलेल्या टॉपिक्सपैकी एक असलेल्यांपुरते मर्यादित करते.

टॉपिक स्पेसिफिकेशन्सची काही उदाहरणे:

- `[]`: कोणत्याही टॉपिक्सना परवानगी आहे.
- `[A]`: पहिल्या स्थानावर A (आणि त्यानंतर काहीही).
- `[null, B]`: पहिल्या स्थानावर काहीही आणि दुसऱ्या स्थानावर B (आणि त्यानंतर काहीही).
- `[A, B]`: पहिल्या स्थानावर A आणि दुसऱ्या स्थानावर B (आणि त्यानंतर काहीही).
- `[[A, B], [A, B]]`: पहिल्या स्थानावर (A किंवा B) आणि दुसऱ्या स्थानावर (A किंवा B) (आणि त्यानंतर काहीही).

### `eth_unsubscribe` {#eth-unsubscribe}

विद्यमान सबस्क्रिप्शन रद्द करते जेणेकरून पुढील कोणत्याही घटना पाठवल्या जाणार नाहीत.

पॅरामीटर्स

1. सबस्क्रिप्शन ID, जसा पूर्वी `eth_subscribe` कॉलवरून परत केला होता.

रिटर्न्स

जर सबस्क्रिप्शन यशस्वीरित्या रद्द केले गेले असेल तर `true`, किंवा दिलेल्या ID सह कोणतेही सबस्क्रिप्शन अस्तित्वात नसल्यास `false`.

उदाहरण:

**विनंती**

```
curl https://eth-mainnet.alchemyapi.io/v2/your-api-key
-X POST
-H "Content-Type: application/json"
-d '{"id": 1, "method": "eth_unsubscribe", "params": ["0x9cef478923ff08bf67fde6c64013158d"]}'
```

**निकाल**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": true
}
```
