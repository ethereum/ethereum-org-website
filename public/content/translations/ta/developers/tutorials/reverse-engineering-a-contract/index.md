## அறிமுகம் {#introduction}

_தொகுதிச்சங்கிலியில் எந்த ரகசியங்களும் இல்லை_, நடக்கும் அனைத்தும் நிலையானவை, சரிபார்க்கக்கூடியவை மற்றும் பொதுவில் கிடைக்கக்கூடியவை. சிறந்த முறையில், [ஒப்பந்தங்கள் அவற்றின் மூலக் குறியீட்டை வெளியிட்டு Etherscan இல் சரிபார்க்கப்பட்டிருக்க வேண்டும்](https://etherscan.io/address/0xb8901acb165ed027e32754e0ffe830802919727f#code). இருப்பினும், [எப்பொழுதும் அப்படி இருப்பதில்லை](https://etherscan.io/address/0x2510c039cc3b061d79e564b38836da87e31b342f#code). இந்தக் கட்டுரையில், மூலக் குறியீடு இல்லாத ஒரு ஒப்பந்தத்தைப் பார்ப்பதன் மூலம் ஒப்பந்தங்களை எவ்வாறு மீள் பொறியியல் செய்வது என்பதை நீங்கள் கற்றுக்கொள்வீர்கள், [`0x2510c039cc3b061d79e564b38836da87e31b342f`](https://etherscan.io/address/0x2510c039cc3b061d79e564b38836da87e31b342f).

மீள் தொகுப்பிகள் (reverse compilers) உள்ளன, ஆனால் அவை எப்போதும் [பயன்படுத்தக்கூடிய முடிவுகளைத்](https://etherscan.io/bytecode-decompiler?a=0x2510c039cc3b061d79e564b38836da87e31b342f) தருவதில்லை. இந்தக் கட்டுரையில், [செயல்பாட்டுக் குறியீடுகளிலிருந்து](https://github.com/wolflo/evm-opcodes) ஒரு ஒப்பந்தத்தை எவ்வாறு கைமுறையாக மீள் பொறியியல் செய்து புரிந்துகொள்வது என்பதையும், ஒரு டிகம்பைலரின் (decompiler) முடிவுகளை எவ்வாறு விளக்குவது என்பதையும் நீங்கள் கற்றுக்கொள்வீர்கள்.

இந்தக் கட்டுரையைப் புரிந்துகொள்ள, நீங்கள் ஏற்கனவே EVM இன் அடிப்படைகளை அறிந்திருக்க வேண்டும், மேலும் EVM அசெம்பிளருடன் (assembler) ஓரளவு பரிச்சயமானவராக இருக்க வேண்டும். [இந்தத் தலைப்புகளைப் பற்றி நீங்கள் இங்கே படிக்கலாம்](https://medium.com/mycrypto/the-ethereum-virtual-machine-how-does-it-work-9abac2b7c9e).

## செயல்படுத்தக்கூடிய குறியீட்டைத் தயார் செய்தல் {#prepare-the-executable-code}

ஒப்பந்தத்திற்கான Etherscan-க்குச் சென்று, **Contract** தாவலைக் கிளிக் செய்து, பின்னர் **Switch to Opcodes View** என்பதைத் தேர்ந்தெடுப்பதன் மூலம் நீங்கள் செயல்பாட்டுக் குறியீடுகளைப் பெறலாம். ஒரு வரிக்கு ஒரு செயல்பாட்டுக் குறியீடு என்ற பார்வையை நீங்கள் பெறுவீர்கள்.

![Opcode View from Etherscan](opcode-view.png)

இருப்பினும், தாவல்களைப் (jumps) புரிந்துகொள்ள, குறியீட்டில் ஒவ்வொரு செயல்பாட்டுக் குறியீடும் எங்கு அமைந்துள்ளது என்பதை நீங்கள் தெரிந்துகொள்ள வேண்டும். அதைச் செய்ய, Google Spreadsheet-ஐத் திறந்து, C நெடுவரிசையில் செயல்பாட்டுக் குறியீடுகளை ஒட்டுவது ஒரு வழியாகும். [ஏற்கனவே தயார் செய்யப்பட்ட இந்த விரிதாளின் நகலை உருவாக்குவதன் மூலம் பின்வரும் படிகளை நீங்கள் தவிர்க்கலாம்](https://docs.google.com/spreadsheets/d/1tKmTJiNjUwHbW64wCKOSJxHjmh0bAUapt6btUYE7kDA/edit?usp=sharing).

அடுத்த படியாக, சரியான குறியீட்டு இருப்பிடங்களைப் பெறுவதன் மூலம் நாம் தாவல்களைப் புரிந்துகொள்ள முடியும். B நெடுவரிசையில் செயல்பாட்டுக் குறியீட்டின் அளவையும், A நெடுவரிசையில் இருப்பிடத்தையும் (பதின்அறுமத்தில் - hexadecimal) வைப்போம். `B1` கலத்தில் இந்தச் சார்பை (function) தட்டச்சு செய்து, குறியீட்டின் இறுதி வரை B நெடுவரிசையின் மீதமுள்ள கலங்களுக்கு அதை நகலெடுத்து ஒட்டவும். இதைச் செய்த பிறகு, நீங்கள் B நெடுவரிசையை மறைக்கலாம்.

```
=1+IF(REGEXMATCH(C1,"PUSH"),REGEXEXTRACT(C1,"PUSH(\d+)"),0)
```

முதலில் இந்தச் சார்பு செயல்பாட்டுக் குறியீட்டிற்காக ஒரு பைட்டைச் சேர்க்கிறது, பின்னர் `PUSH`-ஐத் தேடுகிறது. Push செயல்பாட்டுக் குறியீடுகள் சிறப்பானவை, ஏனெனில் தள்ளப்படும் (pushed) மதிப்பிற்கு அவற்றுக்குக் கூடுதல் பைட்டுகள் தேவைப்படும். செயல்பாட்டுக் குறியீடு `PUSH` ஆக இருந்தால், பைட்டுகளின் எண்ணிக்கையைப் பிரித்தெடுத்து அதைச் சேர்க்கிறோம்.

`A1`-இல் முதல் ஆஃப்செட்டான (offset) சுழியத்தை (zero) வைக்கவும். பின்னர், `A2`-இல், இந்தச் சார்பை வைத்து, A நெடுவரிசையின் மீதமுள்ள கலங்களுக்கு மீண்டும் நகலெடுத்து ஒட்டவும்:

```
=dec2hex(hex2dec(A1)+B1)
```

தாவல்களுக்கு (`JUMP` மற்றும் `JUMPI`) முன்னதாகத் தள்ளப்படும் மதிப்புகள் பதின்அறுமத்தில் (hexadecimal) நமக்கு வழங்கப்படுவதால், இந்தச் சார்பு நமக்கு பதின்அறும மதிப்பை வழங்க வேண்டும்.

## நுழைவுப் புள்ளி (0x00) {#the-entry-point-0x00}

ஒப்பந்தங்கள் எப்போதும் முதல் பைட்டிலிருந்து செயல்படுத்தப்படுகின்றன. இது குறியீட்டின் ஆரம்பப் பகுதியாகும்:

| ஆஃப்செட் | செயல்பாட்டுக் குறியீடு | அடுக்கு (செயல்பாட்டுக் குறியீட்டிற்குப் பிறகு) |
| -----: | ------------ | ------------------------ |
|      0 | PUSH1 0x80   | 0x80                     |
|      2 | PUSH1 0x40   | 0x40, 0x80               |
|      4 | MSTORE       | காலியானது                    |
|      5 | PUSH1 0x04   | 0x04                     |
|      7 | CALLDATASIZE | CALLDATASIZE 0x04        |
|      8 | LT           | CALLDATASIZE\<4           |
|      9 | PUSH2 0x005e | 0x5E CALLDATASIZE\<4      |
|      C | JUMPI        | காலியானது                    |

இந்தக் குறியீடு இரண்டு விஷயங்களைச் செய்கிறது:

1. 0x80 என்பதை 32 பைட் மதிப்பாக 0x40-0x5F நினைவக இடங்களில் எழுதவும் (0x80 என்பது 0x5F இல் சேமிக்கப்படுகிறது, மேலும் 0x40-0x5E அனைத்தும் பூஜ்ஜியங்களாகும்).
2. அழைப்புத் தரவின் அளவைப் படிக்கவும். பொதுவாக ஒரு Ethereum ஒப்பந்தத்திற்கான அழைப்புத் தரவு [ABI (பயன்பாட்டு பைனரி இடைமுகம்)](https://docs.soliditylang.org/en/v0.8.10/abi-spec.html) ஐப் பின்பற்றுகிறது, இதற்குச் செயல்பாட்டுத் தேர்விக்குக் குறைந்தபட்சம் நான்கு பைட்டுகள் தேவைப்படும். அழைப்புத் தரவின் அளவு நான்கிற்கும் குறைவாக இருந்தால், 0x5E க்குத் தாவவும்.

![Flowchart for this portion](flowchart-entry.png)

### 0x5E இல் உள்ள கையாளுநர் (ABI அல்லாத அழைப்புத் தரவிற்கு) {#the-handler-at-0x5e-for-non-abi-call-data}

| ஆஃப்செட் | செயல்பாட்டுக் குறியீடு |
| -----: | ------------ |
|     5E | JUMPDEST     |
|     5F | CALLDATASIZE |
|     60 | PUSH2 0x007c |
|     63 | JUMPI        |

இந்தத் துணுக்கு `JUMPDEST` உடன் தொடங்குகிறது. `JUMPDEST` அல்லாத ஒரு செயல்பாட்டுக் குறியீட்டிற்கு நீங்கள் தாவினால் EVM (Ethereum மெய்நிகர் இயந்திரம்) நிரல்கள் ஒரு விதிவிலக்கை எறியும். பின்னர் அது CALLDATASIZE ஐப் பார்க்கிறது, அது "உண்மை" (அதாவது, பூஜ்ஜியம் அல்ல) எனில் 0x7C க்குத் தாவுகிறது. அதை நாம் கீழே பார்ப்போம்.

| ஆஃப்செட் | செயல்பாட்டுக் குறியீடு | அடுக்கு (செயல்பாட்டுக் குறியீட்டிற்குப் பிறகு)                                                       |
| -----: | ---------- | -------------------------------------------------------------------------- |
|     64 | CALLVALUE  | அழைப்பால் வழங்கப்பட்ட [Wei](/glossary/#wei). Solidity இல் `msg.value` என அழைக்கப்படுகிறது |
|     65 | PUSH1 0x06 | 6 CALLVALUE                                                                |
|     67 | PUSH1 0x00 | 0 6 CALLVALUE                                                              |
|     69 | DUP3       | CALLVALUE 0 6 CALLVALUE                                                    |
|     6A | DUP3       | 6 CALLVALUE 0 6 CALLVALUE                                                  |
|     6B | SLOAD      | Storage[6] CALLVALUE 0 6 CALLVALUE                                         |

எனவே அழைப்புத் தரவு இல்லாதபோது Storage[6] இன் மதிப்பைப் படிக்கிறோம். இந்த மதிப்பு என்னவென்று எங்களுக்கு இன்னும் தெரியவில்லை, ஆனால் அழைப்புத் தரவு இல்லாமல் ஒப்பந்தம் பெற்ற பரிவர்த்தனைகளை நாம் தேடலாம். எந்தவொரு அழைப்புத் தரவும் இல்லாமல் (எனவே எந்த முறையும் இல்லாமல்) ETH ஐப் பரிமாற்றம் செய்யும் பரிவர்த்தனைகள் Etherscan இல் `Transfer` முறையைக் கொண்டுள்ளன. உண்மையில், [ஒப்பந்தம் பெற்ற முதல் பரிவர்த்தனை](https://etherscan.io/tx/0xeec75287a583c36bcc7ca87685ab41603494516a0f5986d18de96c8e630762e7) ஒரு பரிமாற்றம் ஆகும்.

அந்தப் பரிவர்த்தனையைப் பார்த்து, **Click to see More** என்பதைக் கிளிக் செய்தால், உள்ளீட்டுத் தரவு எனப்படும் அழைப்புத் தரவு உண்மையில் காலியாக இருப்பதைக் காண்கிறோம் (`0x`). மதிப்பு 1.559 ETH என்பதையும் கவனியுங்கள், அது பின்னர் பொருத்தமானதாக இருக்கும்.

![The call data is empty](calldata-empty.png)

அடுத்து, **State** தாவலைக் கிளிக் செய்து, நாம் ரிவர்ஸ் இன்ஜினியரிங் செய்யும் ஒப்பந்தத்தை (0x2510...) விரிக்கவும். பரிவர்த்தனையின் போது `Storage[6]` மாறியதை நீங்கள் காணலாம், மேலும் நீங்கள் Hex ஐ **Number** ஆக மாற்றினால், அது 1,559,000,000,000,000,000 ஆக மாறியதைக் காண்பீர்கள், இது wei இல் பரிமாற்றம் செய்யப்பட்ட மதிப்பாகும் (தெளிவுக்காக நான் காற்புள்ளிகளைச் சேர்த்துள்ளேன்), இது அடுத்த ஒப்பந்த மதிப்பிற்கு ஒத்திருக்கிறது.

![Storage[6] இல் மாற்றம்](storage6.png)

[அதே காலகட்டத்தைச் சேர்ந்த பிற `Transfer` பரிவர்த்தனைகளால்](https://etherscan.io/tx/0xf708d306de39c422472f43cb975d97b66fd5d6a6863db627067167cbf93d84d1#statechange) ஏற்பட்ட நிலை மாற்றங்களை நாம் பார்த்தால், `Storage[6]` சிறிது காலத்திற்கு ஒப்பந்தத்தின் மதிப்பைக் கண்காணித்ததைக் காண்கிறோம். இப்போதைக்கு இதை `Value*` என்று அழைப்போம். நட்சத்திரக் குறியீடு (`*`) இந்த மாறி என்ன செய்கிறது என்பது எங்களுக்கு இன்னும் _தெரியாது_ என்பதை நினைவூட்டுகிறது, ஆனால் இது ஒப்பந்த மதிப்பைக் கண்காணிக்க மட்டும் இருக்க முடியாது, ஏனெனில் `ADDRESS BALANCE` ஐப் பயன்படுத்தி உங்கள் கணக்குகளின் இருப்பைப் பெற முடியும் போது, மிகவும் விலையுயர்ந்த சேமிப்பகத்தைப் பயன்படுத்த வேண்டிய அவசியமில்லை. முதல் செயல்பாட்டுக் குறியீடு ஒப்பந்தத்தின் சொந்த முகவரியைத் தள்ளுகிறது. இரண்டாவது அடுக்கின் மேற்புறத்தில் உள்ள முகவரியைப் படித்து, அந்த முகவரியின் இருப்புடன் அதை மாற்றுகிறது.

| ஆஃப்செட் | செயல்பாட்டுக் குறியீடு | அடுக்கு                                                       |

| ஆஃப்செட் | செயல்பாட்டுக் குறியீடு | அடுக்கு                                                                       |

`Value*` ஆனது 2^256-CALLVALUE-1 ஐ விடச் சிறியதாகவோ அல்லது அதற்குச் சமமாகவோ இருந்தால் நாம் தாவுகிறோம். இது ஓவர்ஃப்ளோவைத் தடுப்பதற்கான தர்க்கம் போல் தெரிகிறது. உண்மையில், சில அர்த்தமற்ற செயல்பாடுகளுக்குப் பிறகு (உதாரணமாக, நினைவகத்தில் எழுதுவது நீக்கப்பட உள்ளது) ஆஃப்செட் 0x01DE இல் ஓவர்ஃப்ளோ கண்டறியப்பட்டால் ஒப்பந்தம் மீளமைக்கப்படுவதை நாம் காண்கிறோம், இது சாதாரண நடத்தை ஆகும்.

அத்தகைய ஓவர்ஃப்ளோ ஏற்படுவது மிகவும் சாத்தியமற்றது என்பதைக் கவனியுங்கள், ஏனெனில் இதற்கு அழைப்பு மதிப்பு மற்றும் `Value*` ஆகியவை 2^256 wei, அதாவது சுமார் 10^59 ETH உடன் ஒப்பிடக்கூடியதாக இருக்க வேண்டும். [இதை எழுதும் நேரத்தில், மொத்த ETH வழங்கல் இருநூறு மில்லியனுக்கும் குறைவாகவே உள்ளது](https://etherscan.io/stat/supply).

|    1E3 | JUMP     |

நாம் இங்கு வந்திருந்தால், `Value* + CALLVALUE` ஐப் பெற்று ஆஃப்செட் 0x75 க்குத் தாவவும்.

| ஆஃப்செட் | செயல்பாட்டுக் குறியீடு | அடுக்கு                           |

நாம் இங்கு வந்தால் (இதற்கு அழைப்புத் தரவு காலியாக இருக்க வேண்டும்) `Value*` உடன் அழைப்பு மதிப்பைச் சேர்க்கிறோம். இது `Transfer` பரிவர்த்தனைகள் என்ன செய்கின்றன என்று நாம் கூறுவதுடன் ஒத்துப்போகிறது.

இறுதியாக, அடுக்கை அழிக்கவும் (இது அவசியமில்லை) மற்றும் பரிவர்த்தனையின் வெற்றிகரமான முடிவைக் குறிக்கவும்.

சுருக்கமாகச் சொல்வதானால், ஆரம்பக் குறியீட்டிற்கான பாய்வு வரைபடம் இங்கே உள்ளது.

## 0x7C இல் உள்ள கையாளுநர் {#the-handler-at-0x7c}

இந்தக் கையாளுநர் என்ன செய்கிறது என்பதை நான் வேண்டுமென்றே தலைப்பில் குறிப்பிடவில்லை. இந்த குறிப்பிட்ட ஒப்பந்தம் எவ்வாறு செயல்படுகிறது என்பதை உங்களுக்குக் கற்பிப்பது இதன் நோக்கமல்ல, மாறாக ஒப்பந்தங்களை எவ்வாறு பின்னோக்கிப் பொறியியல் (reverse engineer) செய்வது என்பதைக் கற்பிப்பதே ஆகும். குறியீட்டைப் பின்பற்றுவதன் மூலம், நான் கற்றுக்கொண்ட அதே வழியில் இது என்ன செய்கிறது என்பதை நீங்களும் கற்றுக்கொள்வீர்கள்.

நாம் பல இடங்களிலிருந்து இங்கு வருகிறோம்:

- 1, 2 அல்லது 3 பைட்டுகள் கொண்ட அழைப்புத் தரவு இருந்தால் (ஆஃப்செட் 0x63 இலிருந்து)
- முறை கையொப்பம் (method signature) தெரியவில்லை என்றால் (ஆஃப்செட்டுகள் 0x42 மற்றும் 0x5D இலிருந்து)

| ஆஃப்செட் | செயல்பாட்டுக் குறியீடு | அடுக்கு |
| -----: | ------------ | -------------------- |

இது மற்றொரு சேமிப்பக அறை (storage cell) ஆகும், இதை என்னால் எந்தப் பரிவர்த்தனைகளிலும் கண்டுபிடிக்க முடியவில்லை, எனவே இதன் அர்த்தம் என்ன என்பதை அறிவது கடினம். கீழே உள்ள குறியீடு இதை மேலும் தெளிவாக்கும்.

இப்போது விஷயங்கள் மிகவும் தெளிவாக உள்ளன. இந்த ஒப்பந்தம் ஒரு [பதிலியாக (proxy)](https://blog.openzeppelin.com/proxy-patterns/) செயல்பட முடியும், உண்மையான வேலையைச் செய்ய Storage[3] இல் உள்ள முகவரியை அழைக்கிறது. `DELEGATE_CALL` ஒரு தனி ஒப்பந்தத்தை அழைக்கிறது, ஆனால் அதே சேமிப்பகத்தில் தங்கியுள்ளது. இதன் பொருள், நாம் எதற்குப் பதிலியாக இருக்கிறோமோ அந்த ஒப்படைக்கப்பட்ட ஒப்பந்தம், அதே சேமிப்பக இடத்தையே அணுகுகிறது. அழைப்பிற்கான அளவுருக்கள் (parameters) பின்வருமாறு:

- _எரிவாயு (Gas)_: மீதமுள்ள அனைத்து எரிவாயுவும்
- _அழைக்கப்பட்ட முகவரி_: Storage[3]-முகவரியாக

எனவே இந்தக் குறியீடு அடுக்கில் (X) ஒரு நினைவகச் சுட்டியைப் பெறுகிறது, மேலும் இது 0x80 - X என்ற இடையகத்துடன் ஒப்பந்தத்தை `RETURN` செய்யச் செய்கிறது.

`splitter()` ஐப் பொறுத்தவரை, இது நாம் எந்த முகவரிக்குப் பதிலியாக இருக்கிறோமோ அந்த முகவரியை வழங்குகிறது. `RETURN` ஆனது 0x80-0x9F இல் இடையகத்தை வழங்குகிறது, இங்குதான் நாம் இந்தத் தரவை எழுதினோம் (மேலே உள்ள ஆஃப்செட் 0x130).

ஆனால் வேறு எந்தச் செயல்பாடும் Storage[3] இல் உள்ள ஒப்பந்தத்தால் வழங்கப்படுகிறது என்பதை நாம் அறிவோம். அந்த ஒப்பந்தம் என்னவென்று நமக்குத் தெரிந்தால், அது நமக்கு ஒரு துப்பைக் கொடுக்கலாம். அதிர்ஷ்டவசமாக, இது தொகுதிச்சங்கிலி மற்றும் குறைந்தபட்சம் கோட்பாட்டளவிலாவது அனைத்தும் அறியப்பட்டவை. Storage[3] ஐ அமைக்கும் எந்த முறைகளையும் நாம் பார்க்கவில்லை, எனவே அது ஆக்கியால் அமைக்கப்பட்டிருக்க வேண்டும்.

## பதிலி ஒப்பந்தம் {#the-proxy-contract}

மேலே உள்ள அசல் ஒப்பந்தத்திற்கு நாம் பயன்படுத்திய அதே நுட்பங்களைப் பயன்படுத்தி, பின்வரும் நிலைகளில் ஒப்பந்தம் மீளமைக்கப்படுவதை நாம் காணலாம்:

மீதமுள்ள முறைகளில் ஒன்று `claim(<params>)`, மற்றொன்று `isClaimed(<params>)`, எனவே இது ஒரு ஏர்ட்ராப் ஒப்பந்தம் போல் தெரிகிறது. மீதமுள்ளவற்றை செயல்பாட்டுக் குறியீடு வாரியாகப் பார்ப்பதற்குப் பதிலாக, நாம் [டிகம்பைலரை முயற்சிக்கலாம்](https://etherscan.io/bytecode-decompiler?a=0x2f81e57ff4f4d83b40a9f719fd892d8e806e0761), இது இந்த ஒப்பந்தத்திலிருந்து மூன்று சார்புகளுக்குப் பயன்படுத்தக்கூடிய முடிவுகளை உருவாக்குகிறது. மற்றவற்றை ரிவர்ஸ் இன்ஜினியரிங் செய்வது வாசகர்களுக்கான பயிற்சியாக விடப்பட்டுள்ளது.

### scaleAmountByPercentage {#scaleamountbypercentage}

இந்தச் சார்புக்கு டிகம்பைலர் நமக்கு வழங்குவது இதுதான்:

```python
def unknown8ffb5c97(uint256 _param1, uint256 _param2) payable:
  require calldata.size - 4 >=′ 64
  if _param1 and _param2 > -1 / _param1:
      revert with 0, 17
  return (_param1 * _param2 / 100 * 10^6)
```

முதல் `require` ஆனது, சார்பு கையொப்பத்தின் நான்கு பைட்டுகளுக்கு மேலதிகமாக, அழைப்புத் தரவில் இரண்டு அளவுருக்களுக்குப் போதுமான குறைந்தது 64 பைட்டுகள் உள்ளதா எனச் சோதிக்கிறது. இல்லையெனில், வெளிப்படையாக ஏதோ தவறு உள்ளது.

`if` கூற்று `_param1` பூஜ்ஜியமாக இல்லை என்பதையும், `_param1 * _param2` எதிர்மறையாக இல்லை என்பதையும் சரிபார்ப்பதாகத் தெரிகிறது. இது அநேகமாக ரேப் அரவுண்ட் (wrap around) நிகழ்வுகளைத் தடுப்பதற்காக இருக்கலாம்.

இறுதியாக, சார்பு அளவிடப்பட்ட மதிப்பை வழங்குகிறது.

### claim {#claim}

டிகம்பைலர் உருவாக்கும் குறியீடு சிக்கலானது, மேலும் அவை அனைத்தும் நமக்குத் தொடர்புடையவை அல்ல. பயனுள்ள தகவல்களை வழங்குவதாக நான் நம்பும் வரிகளில் கவனம் செலுத்த, அவற்றில் சிலவற்றை நான் தவிர்க்கப் போகிறேன்

```python
def unknown2e7ba6ef(uint256 _param1, uint256 _param2, uint256 _param3, array _param4) payable:
  ...
  require _param2 == addr(_param2)
  ...
  if currentWindow <= _param1:
      revert with 0, 'cannot claim for a future window'
```

இங்கே நாம் இரண்டு முக்கியமான விஷயங்களைக் காண்கிறோம்:

- `_param2` என்பது `uint256` ஆக அறிவிக்கப்பட்டிருந்தாலும், அது உண்மையில் ஒரு முகவரி
- `_param1` என்பது உரிமைக்கோரப்படும் சாளரம் (window), இது `currentWindow` அல்லது அதற்கு முந்தையதாக இருக்க வேண்டும்.

```python
  ...
  if stor5[_claimWindow][addr(_claimFor)]:
      revert with 0, 'Account already claimed the given window'
```

எனவே இப்போது Storage[5] என்பது சாளரங்கள் மற்றும் முகவரிகளின் அணிவரிசை (array) என்பதையும், அந்த முகவரி அந்தச் சாளரத்திற்கான வெகுமதியை உரிமைக்கோரியதா என்பதையும் நாம் அறிவோம்.

```python
  ...
  idx = 0
  s = 0
  while idx < _param4.length:
  ...
      if s + sha3(mem[(32 * _param4.length) + 328 len mem[(32 * _param4.length) + 296]]) > mem[(32 * idx) + 296]:
          mem[mem[64] + 32] = mem[(32 * idx) + 296]
          ...
          s = sha3(mem[_62 + 32 len mem[_62]])
          continue
      ...
      s = sha3(mem[_66 + 32 len mem[_66]])
      continue
  if unknown2eb4a7ab != s:
      revert with 0, 'Invalid proof'
```

`unknown2eb4a7ab` என்பது உண்மையில் `merkleRoot()` சார்பு என்பதை நாம் அறிவோம், எனவே இந்தக் குறியீடு ஒரு [மெர்கல் சான்றைச்](https://medium.com/crypto-0-nite/merkle-proofs-explained-6dd429623dc5) சரிபார்ப்பது போல் தெரிகிறது. இதன் பொருள் `_param4` என்பது ஒரு மெர்கல் சான்று என்பதாகும்.

```python
  call addr(_param2) with:
     value unknown81e580d3[_param1] * _param3 / 100 * 10^6 wei
       gas 30000 wei
```

ஒரு ஒப்பந்தம் தனது சொந்த ETH-ஐ மற்றொரு முகவரிக்கு (ஒப்பந்தம் அல்லது வெளிப்புறமாகச் சொந்தமான கணக்கு) இப்படித்தான் பரிமாற்றம் செய்கிறது. பரிமாற்றம் செய்யப்பட வேண்டிய தொகையான மதிப்புடன் அது அதை அழைக்கிறது. எனவே இது ETH-இன் ஏர்ட்ராப் போல் தெரிகிறது.


### 1e7df9d3 {#1e7df9d3}

இந்தச் சார்பு மேலே உள்ள [`claim`](#claim)-ஐப் போலவே உள்ளது. இது ஒரு மெர்கல் சான்றையும் சரிபார்க்கிறது, முதலாவதாக ETH-ஐப் பரிமாற்றம் செய்ய முயற்சிக்கிறது, மேலும் அதே வகையான பதிவு உள்ளீட்டை உருவாக்குகிறது.

```python
def unknown1e7df9d3(uint256 _param1, uint256 _param2, array _param3) payable:
```
