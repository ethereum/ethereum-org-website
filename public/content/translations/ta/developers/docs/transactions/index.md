பரிவர்த்தனைகள் என்பவை கணக்குகளிலிருந்து வரும் கிரிப்டோகிராஃபிக் முறையில் கையொப்பமிடப்பட்ட வழிமுறைகளாகும். [எத்திரியம்](/) பிணையத்தின் நிலையைப் புதுப்பிக்க ஒரு கணக்கு ஒரு பரிவர்த்தனையைத் தொடங்கும். ஒரு கணக்கிலிருந்து மற்றொரு கணக்கிற்கு ETH-ஐப் பரிமாற்றம் செய்வதே எளிமையான பரிவர்த்தனையாகும்.

## முன்நிபந்தனைகள் {#prerequisites}

இந்தப் பக்கத்தை நீங்கள் நன்கு புரிந்துகொள்ள, முதலில் [கணக்குகள்](/developers/docs/accounts/) மற்றும் எங்களின் [எத்திரியம் அறிமுகம்](/developers/docs/intro-to-ethereum/) ஆகியவற்றைப் படிக்குமாறு பரிந்துரைக்கிறோம்.

## பரிவர்த்தனை என்றால் என்ன? {#whats-a-transaction}

எத்திரியம் பரிவர்த்தனை என்பது வெளிப்புறமாகச் சொந்தமான கணக்கால் தொடங்கப்பட்ட ஒரு செயலைக் குறிக்கிறது, வேறுவிதமாகக் கூறினால், இது ஒரு ஒப்பந்தத்தால் அல்லாமல் ஒரு மனிதனால் நிர்வகிக்கப்படும் கணக்காகும். எடுத்துக்காட்டாக, பாப் ஆலிஸுக்கு 1 ETH-ஐ அனுப்பினால், பாப்பின் கணக்கில் பற்று வைக்கப்பட வேண்டும் மற்றும் ஆலிஸின் கணக்கில் வரவு வைக்கப்பட வேண்டும். நிலையை மாற்றும் இந்தச் செயல் ஒரு பரிவர்த்தனைக்குள் நிகழ்கிறது.

_[Ethereum EVM illustrated](https://takenobu-hs.github.io/downloads/ethereum_evm_illustrated.pdf) என்பதிலிருந்து தழுவப்பட்ட வரைபடம்_

EVM-இன் நிலையை மாற்றும் பரிவர்த்தனைகள், முழுப் பிணையத்திற்கும் ஒளிபரப்பப்பட வேண்டும். EVM-இல் ஒரு பரிவர்த்தனையைச் செயல்படுத்துவதற்கான கோரிக்கையை எந்தவொரு கணுவும் ஒளிபரப்பலாம்; இது நடந்த பிறகு, ஒரு சரிபார்ப்பான் பரிவர்த்தனையைச் செயல்படுத்தி, அதன் விளைவாக ஏற்படும் நிலை மாற்றத்தைப் பிணையத்தின் மற்ற பகுதிகளுக்குப் பரப்பும்.

பரிவர்த்தனைகளுக்கு ஒரு கட்டணம் தேவை மற்றும் அவை சரிபார்க்கப்பட்ட தொகுதியில் சேர்க்கப்பட வேண்டும். இந்த மேலோட்டத்தை எளிமையாக்க, எரிவாயு கட்டணம் மற்றும் சரிபார்ப்பை வேறு இடத்தில் காண்போம்.

சமர்ப்பிக்கப்பட்ட பரிவர்த்தனையில் பின்வரும் தகவல்கள் அடங்கும்:

- `from` – பரிவர்த்தனையில் கையொப்பமிடும் அனுப்புநரின் முகவரி. ஒப்பந்தக் கணக்குகளால் பரிவர்த்தனைகளை அனுப்ப முடியாது என்பதால் இது வெளிப்புறமாகச் சொந்தமான கணக்காக இருக்கும்
- `to` – பெறும் முகவரி (வெளிப்புறமாகச் சொந்தமான கணக்காக இருந்தால், பரிவர்த்தனை மதிப்பை மாற்றும். ஒப்பந்தக் கணக்காக இருந்தால், பரிவர்த்தனை ஒப்பந்தக் குறியீட்டைச் செயல்படுத்தும்)
- `signature` – அனுப்புநரின் அடையாளங்காட்டி. அனுப்புநரின் தனிப்பட்ட திறவுகோல் பரிவர்த்தனையில் கையொப்பமிடும்போது இது உருவாக்கப்படுகிறது மற்றும் அனுப்புநர் இந்தப் பரிவர்த்தனைக்கு அங்கீகாரம் அளித்துள்ளார் என்பதை உறுதிப்படுத்துகிறது
- `nonce` - கணக்கிலிருந்து பரிவர்த்தனை எண்ணைக் குறிக்கும் தொடர்ச்சியாக அதிகரிக்கும் கவுண்டர் (நான்ஸ்)
- `value` – அனுப்புநரிடமிருந்து பெறுநருக்குப் பரிமாற்றம் செய்ய வேண்டிய ETH-இன் அளவு (WEI-இல் குறிப்பிடப்படுகிறது, இங்கு 1ETH என்பது 1e+18wei-க்குச் சமம்)
- `input data` – தன்னிச்சையான தரவைச் சேர்ப்பதற்கான விருப்பப் புலம்
- `gasLimit` – பரிவர்த்தனையால் நுகரப்படக்கூடிய அதிகபட்ச எரிவாயு அலகுகளின் அளவு. ஒவ்வொரு கணக்கீட்டுப் படிக்கும் தேவையான எரிவாயு அலகுகளை [EVM](/developers/docs/evm/opcodes) குறிப்பிடுகிறது
- `maxPriorityFeePerGas` - சரிபார்ப்பானுக்கு முன்னுரிமைக் கட்டணமாகச் சேர்க்கப்பட வேண்டிய நுகரப்பட்ட எரிவாயுவின் அதிகபட்ச விலை
- `maxFeePerGas` - பரிவர்த்தனைக்குச் செலுத்தத் தயாராக உள்ள ஒரு யூனிட் எரிவாயுவிற்கான அதிகபட்ச கட்டணம் (`baseFeePerGas` மற்றும் `maxPriorityFeePerGas` உட்பட)

எரிவாயு என்பது ஒரு சரிபார்ப்பானால் பரிவர்த்தனையைச் செயலாக்கத் தேவையான கணக்கீட்டைக் குறிக்கிறது. இந்தக் கணக்கீட்டிற்குப் பயனர்கள் கட்டணம் செலுத்த வேண்டும். `gasLimit` மற்றும் `maxPriorityFeePerGas` ஆகியவை சரிபார்ப்பானுக்குச் செலுத்தப்படும் அதிகபட்ச பரிவர்த்தனை கட்டணத்தைத் தீர்மானிக்கின்றன. [எரிவாயு பற்றி மேலும் அறிய](/developers/docs/gas/).

பரிவர்த்தனைப் பொருள் சற்று இதுபோல இருக்கும்:

```js
{
  from: "0xEA674fdDe714fd979de3EdF0F56AA9716B898ec8",
  to: "0xac03bb73b6a9e108530aff4df5077c2b3d481e5a",
  gasLimit: "21000",
  maxFeePerGas: "300",
  maxPriorityFeePerGas: "10",
  nonce: "0",
  value: "10000000000"
}
```

ஆனால் ஒரு பரிவர்த்தனைப் பொருள் அனுப்புநரின் தனிப்பட்ட திறவுகோலைப் பயன்படுத்திக் கையொப்பமிடப்பட வேண்டும். பரிவர்த்தனை அனுப்புநரிடமிருந்து மட்டுமே வந்திருக்க முடியும் என்பதையும், மோசடியாக அனுப்பப்படவில்லை என்பதையும் இது நிரூபிக்கிறது.

கோ எத்திரியம் (கெத்) போன்ற ஒரு எத்திரியம் கிளையண்ட் இந்தக் கையொப்பமிடுதல் செயல்முறையைக் கையாளும்.

எடுத்துக்காட்டு [ஜேசன்-ஆர்பிசி](/developers/docs/apis/json-rpc) அழைப்பு:

```json
{
  "id": 2,
  "jsonrpc": "2.0",
  "method": "account_signTransaction",
  "params": [
    {
      "from": "0x1923f626bb8dc025849e00f99c25fe2b2f7fb0db",
      "gas": "0x55555",
      "maxFeePerGas": "0x1234",
      "maxPriorityFeePerGas": "0x1234",
      "input": "0xabcd",
      "nonce": "0x0",
      "to": "0x07a565b7ed7d7a678680a4c162885bedbb695fe0",
      "value": "0x1234"
    }
  ]
}
```

எடுத்துக்காட்டுப் பதில்:

```json
{

  "id": 2,

    "tx": {
      "nonce": "0x0",
      "maxFeePerGas": "0x1234",
      "maxPriorityFeePerGas": "0x1234",
      "gas": "0x55555",
```

- `raw` என்பது [Recursive Length Prefix (RLP)](/developers/docs/data-structures-and-encoding/rlp) குறியிடப்பட்ட வடிவத்தில் கையொப்பமிடப்பட்ட பரிவர்த்தனையாகும்
- `tx` என்பது JSON வடிவத்தில் கையொப்பமிடப்பட்ட பரிவர்த்தனையாகும்

கையொப்ப ஹாஷ் மூலம், பரிவர்த்தனை அனுப்புநரிடமிருந்து வந்தது மற்றும் பிணையத்தில் சமர்ப்பிக்கப்பட்டது என்பதைக் கிரிப்டோகிராஃபிக் முறையில் நிரூபிக்க முடியும்.

### தரவுப் புலம் {#the-data-field}

பெரும்பாலான பரிவர்த்தனைகள் வெளிப்புறமாகச் சொந்தமான கணக்கிலிருந்து ஒரு ஒப்பந்தத்தை அணுகுகின்றன.
பெரும்பாலான ஒப்பந்தங்கள் Solidity-இல் எழுதப்பட்டுள்ளன மற்றும் அவற்றின் தரவுப் புலத்தை [பயன்பாட்டு பைனரி இடைமுகத்திற்கு (ABI)](/glossary/#abi) ஏற்ப விளக்குகின்றன.

முதல் நான்கு பைட்டுகள், செயல்பாட்டின் பெயர் மற்றும் வாதங்களின் ஹாஷைப் பயன்படுத்தி எந்தச் செயல்பாட்டை அழைக்க வேண்டும் என்பதைக் குறிப்பிடுகின்றன.
[இந்தத் தரவுத்தளத்தைப்](https://www.4byte.directory/signatures/) பயன்படுத்தித் தேர்வியிலிருந்து செயல்பாட்டை நீங்கள் சில நேரங்களில் அடையாளம் காண முடியும்.

மீதமுள்ள அழைப்புத் தரவு வாதங்களாகும், அவை [ABI விவரக்குறிப்புகளில் குறிப்பிட்டுள்ளபடி குறியிடப்பட்டுள்ளன](https://docs.soliditylang.org/en/latest/abi-spec.html#formal-specification-of-the-encoding).

எடுத்துக்காட்டாக, [இந்தப் பரிவர்த்தனையைப்](https://etherscan.io/tx/0xd0dcbe007569fcfa1902dae0ab8b4e078efe42e231786312289b1eee5590f6a1) பார்ப்போம்.

ABI விவரக்குறிப்புகளின்படி, முழு எண் மதிப்புகள் (20-பைட் முழு எண்களான முகவரிகள் போன்றவை) ABI-இல் 32-பைட் சொற்களாகத் தோன்றும், முன்புறம் பூஜ்ஜியங்களால் நிரப்பப்பட்டிருக்கும்.
எனவே `to` முகவரி [`4f6742badb049791cd9a37ea913f2bac38d01279`](https://etherscan.io/address/0x4f6742badb049791cd9a37ea913f2bac38d01279) என்பதை நாம் அறிவோம்.
`value` என்பது 0x3b0559f4 = 990206452 ஆகும்.

### பரிவர்த்தனை விவரிப்பான்கள் {#transaction-descriptors}

தரவுப் புலத்தில் ஒளிபுகா ஹெக்ஸாடெசிமல் பைட்டுகள் இருப்பதால், ஒரு பரிவர்த்தனை உண்மையில் என்ன செயலைச் செய்யும் என்பதைச் சரிபார்ப்பது மிகவும் கடினமாக இருக்கும். இந்த "குருட்டுக் கையொப்பமிடுதல்" பாதிப்பு, [பரிவர்த்தனை விவரிப்பான்களைப்](https://eips.ethereum.org/EIPS/eip-7730) (ERC-7730 மூலம் வரையறுக்கப்பட்டது) பயன்படுத்துவதன் மூலம் **[தெளிவான கையொப்பமிடுதல் (Clear Signing)](https://clearsigning.org/)** மூலம் தீர்க்கப்படுகிறது.  

ERC-7730 விவரக்குறிப்பு, ABI-கள் மற்றும் EVM பரிவர்த்தனை அழைப்புத் தரவு, EIP-712 செய்திகள் மற்றும் EIP-4337 பயனர் செயல்பாடுகள் போன்ற கட்டமைக்கப்பட்ட செய்திகளில் காணப்படும் தரவை மேம்படுத்தப் பரிவர்த்தனை விவரிப்பான்களைப் (பெரும்பாலும் JSON கோப்புகளாகக் கட்டமைக்கப்பட்டுள்ளது) பயன்படுத்துகிறது. டெவலப்பர்கள் இந்த விவரிப்பான்களைப் பயன்படுத்தி குறிப்பிட்ட பரிவர்த்தனை மாறிகளை நேரடியாக வடிவமைக்கும் டெம்ப்ளேட்டுகளுக்கு வரைபடமாக்குகிறார்கள், இதன் மூலம் அடிப்படைத் தரவு பயன்பாடுகளுக்கு இயந்திரம் படிக்கக்கூடியதாக இருப்பதை உறுதிசெய்கிறார்கள்.

முன்பகுதியில், வாலெட்டுகள் இந்த வடிவமைக்கும் சூழலைப் பயன்படுத்தி ஒளிபுகா பைட் குறியீட்டைத் தெளிவான, மனிதர்கள் படிக்கக்கூடிய தகவலாக மொழிபெயர்க்கின்றன. வில்லை முகவரிகள் போன்ற மதிப்புகளை அங்கீகரிக்கப்பட்ட டிக்கர்களாகவோ அல்லது தொகைகளை தசமங்களாகவோ தானாகவே தீர்ப்பதன் மூலம், பயனர்கள் கையொப்பமிடுவதற்கு முன்பு பரிவர்த்தனையின் சரியான நோக்கத்தின் எளிய மொழிச் சுருக்கம் (எ.கா., 'குறைந்தது 0.25 WETH-க்கு 1000 USDC-ஐப் பரிமாற்றம் செய்') வழங்கப்படுகிறது.

## பரிவர்த்தனைகளின் வகைகள் {#types-of-transactions}

எத்திரியத்தில் சில வெவ்வேறு வகையான பரிவர்த்தனைகள் உள்ளன:

- வழக்கமான பரிவர்த்தனைகள்: ஒரு கணக்கிலிருந்து மற்றொரு கணக்கிற்கான பரிவர்த்தனை.
- ஒப்பந்த நிலைநிறுத்தப் பரிவர்த்தனைகள்: 'to' முகவரி இல்லாத பரிவர்த்தனை, இதில் தரவுப் புலம் ஒப்பந்தக் குறியீட்டிற்குப் பயன்படுத்தப்படுகிறது.
- ஒப்பந்தத்தைச் செயல்படுத்துதல்: நிலைநிறுத்தப்பட்ட திறன் ஒப்பந்தத்துடன் தொடர்பு கொள்ளும் பரிவர்த்தனை. இந்த நிலையில், 'to' முகவரி என்பது திறன் ஒப்பந்த முகவரியாகும்.

### எரிவாயு பற்றி {#on-gas}

குறிப்பிட்டுள்ளபடி, பரிவர்த்தனைகளைச் செயல்படுத்த [எரிவாயு](/developers/docs/gas/) செலவாகும். எளிய பரிமாற்றப் பரிவர்த்தனைகளுக்கு 21000 யூனிட் எரிவாயு தேவைப்படுகிறது.

எனவே பாப் ஆலிஸுக்கு 1 ETH-ஐ 190 gwei `baseFeePerGas` மற்றும் 10 gwei `maxPriorityFeePerGas`-இல் அனுப்ப, பாப் பின்வரும் கட்டணத்தைச் செலுத்த வேண்டும்:

```
(190 + 10) * 21000 = 4,200,000 gwei
--அல்லது--
0.0042 ETH
```

பாப்பின் கணக்கில் **-1.0042 ETH** பற்று வைக்கப்படும் (ஆலிஸுக்கு 1 ETH + எரிவாயு கட்டணத்தில் 0.0042 ETH)

ஆலிஸின் கணக்கில் **+1.0 ETH** வரவு வைக்கப்படும்

அடிப்படை கட்டணம் எரிக்கப்படும் **-0.00399 ETH**

சரிபார்ப்பான் முன்னுரிமைக் கட்டணத்தை வைத்துக்கொள்வார் **+0.000210 ETH**

### திறன் ஒப்பந்தப் பரிமாற்றங்கள் {#smart-contract-interactions}

திறன் ஒப்பந்தம் சம்பந்தப்பட்ட எந்தவொரு பரிவர்த்தனைக்கும் எரிவாயு தேவை.

திறன் ஒப்பந்தங்களில் [`view`](https://docs.soliditylang.org/en/latest/contracts.html#view-functions) அல்லது [`pure`](https://docs.soliditylang.org/en/latest/contracts.html#pure-functions) செயல்பாடுகள் எனப்படும் செயல்பாடுகளும் இருக்கலாம், அவை ஒப்பந்தத்தின் நிலையை மாற்றாது. எனவே, EOA-இலிருந்து இந்தச் செயல்பாடுகளை அழைப்பதற்கு எந்த எரிவாயுவும் தேவையில்லை. இந்தச் சூழலுக்கான அடிப்படை RPC அழைப்பு [`eth_call`](/developers/docs/apis/json-rpc#eth_call) ஆகும்.

`eth_call`-ஐப் பயன்படுத்தி அணுகுவதைப் போலல்லாமல், இந்த `view` அல்லது `pure` செயல்பாடுகள் பொதுவாக உள்நாட்டிலும் (அதாவது, ஒப்பந்தத்திலிருந்தோ அல்லது மற்றொரு ஒப்பந்தத்திலிருந்தோ) அழைக்கப்படுகின்றன, இதற்கு எரிவாயு செலவாகும்.

## பரிவர்த்தனை வாழ்க்கைச் சுழற்சி {#transaction-lifecycle}

பரிவர்த்தனை சமர்ப்பிக்கப்பட்டதும் பின்வருபவை நிகழும்:

1. ஒரு பரிவர்த்தனை ஹாஷ் கிரிப்டோகிராஃபிக் முறையில் உருவாக்கப்படுகிறது:
   `0x97d99bc7729211111a21b12c933c949d4f31684f1d6954ff477d0477538ff017`
2. பரிவர்த்தனை பின்னர் பிணையத்திற்கு ஒளிபரப்பப்படுகிறது மற்றும் நிலுவையில் உள்ள மற்ற அனைத்துப் பிணையப் பரிவர்த்தனைகளையும் கொண்ட பரிவர்த்தனைத் தொகுப்பில் சேர்க்கப்படுகிறது.
3. பரிவர்த்தனையைச் சரிபார்த்து அதை "வெற்றிகரமானது" எனக் கருத, ஒரு சரிபார்ப்பான் உங்கள் பரிவர்த்தனையைத் தேர்ந்தெடுத்து அதை ஒரு தொகுதியில் சேர்க்க வேண்டும்.
4. நேரம் செல்லச் செல்ல உங்கள் பரிவர்த்தனையைக் கொண்ட தொகுதி "நியாயப்படுத்தப்பட்டது" பின்னர் "இறுதியாக்கப்பட்டது" என மேம்படுத்தப்படும். இந்த மேம்பாடுகள் உங்கள் பரிவர்த்தனை வெற்றிகரமானது மற்றும் ஒருபோதும் மாற்றப்படாது என்பதை மிகவும் உறுதியாக்குகின்றன. ஒரு தொகுதி "இறுதியாக்கப்பட்டது" என்றால், பல பில்லியன் டாலர்கள் செலவாகும் பிணைய அளவிலான தாக்குதலால் மட்டுமே அதை மாற்ற முடியும்.

## தட்டச்சு செய்யப்பட்ட பரிவர்த்தனை உறை {#typed-transaction-envelope}

எத்திரியம் முதலில் பரிவர்த்தனைகளுக்கு ஒரு வடிவமைப்பைக் கொண்டிருந்தது. ஒவ்வொரு பரிவர்த்தனையிலும் நான்ஸ், எரிவாயு விலை, எரிவாயு வரம்பு, பெறும் முகவரி, மதிப்பு, தரவு, v, r மற்றும் s ஆகியவை இருந்தன. இந்தப் புலங்கள் [RLP-குறியிடப்பட்டவை](/developers/docs/data-structures-and-encoding/rlp/), இதுபோல இருக்கும்:

`RLP([nonce, gasPrice, gasLimit, to, value, data, v, r, s])`

பழைய பரிவர்த்தனை வடிவங்களைப் பாதிக்காமல் அணுகல் பட்டியல்கள் மற்றும் [EIP-1559](https://eips.ethereum.org/EIPS/eip-1559) போன்ற புதிய அம்சங்களைச் செயல்படுத்த அனுமதிக்கும் வகையில் பல வகையான பரிவர்த்தனைகளை ஆதரிக்க எத்திரியம் உருவாகியுள்ளது.

[EIP-2718](https://eips.ethereum.org/EIPS/eip-2718) இந்த நடத்தைக்கு அனுமதிக்கிறது. பரிவர்த்தனைகள் இவ்வாறு விளக்கப்படுகின்றன:

`TransactionType || TransactionPayload`

இங்குப் புலங்கள் இவ்வாறு வரையறுக்கப்பட்டுள்ளன:

- `TransactionType` - 0 மற்றும் 0x7f-க்கு இடையிலான எண், மொத்தம் 128 சாத்தியமான பரிவர்த்தனை வகைகளுக்கு.
- `TransactionPayload` - பரிவர்த்தனை வகையால் வரையறுக்கப்பட்ட தன்னிச்சையான பைட் வரிசை.

`TransactionType` மதிப்பின் அடிப்படையில், ஒரு பரிவர்த்தனையை இவ்வாறு வகைப்படுத்தலாம்:

1. **வகை 0 (பழைய) பரிவர்த்தனைகள்:** எத்திரியம் தொடங்கப்பட்டதிலிருந்து பயன்படுத்தப்படும் அசல் பரிவர்த்தனை வடிவம். திறன் ஒப்பந்தங்களுக்கான டைனமிக் எரிவாயு கட்டணக் கணக்கீடுகள் அல்லது அணுகல் பட்டியல்கள் போன்ற [EIP-1559](https://eips.ethereum.org/EIPS/eip-1559)-இன் அம்சங்கள் அவற்றில் இல்லை. பழைய பரிவர்த்தனைகள் அவற்றின் வரிசைப்படுத்தப்பட்ட வடிவத்தில் அவற்றின் வகையைக் குறிக்கும் குறிப்பிட்ட முன்னொட்டைக் கொண்டிருக்கவில்லை, [Recursive Length Prefix (RLP)](/developers/docs/data-structures-and-encoding/rlp) குறியாக்கத்தைப் பயன்படுத்தும்போது `0xf8` பைட்டுடன் தொடங்குகின்றன. இந்தப் பரிவர்த்தனைகளுக்கான TransactionType மதிப்பு `0x0` ஆகும்.

2. **வகை 1 பரிவர்த்தனைகள்:** எத்திரியத்தின் [பெர்லின் மேம்பாட்டின்](/ethereum-forks/#berlin) ஒரு பகுதியாக [EIP-2930](https://eips.ethereum.org/EIPS/eip-2930)-இல் அறிமுகப்படுத்தப்பட்ட இந்தப் பரிவர்த்தனைகளில் `accessList` அளவுரு அடங்கும். இந்தப் பட்டியல் பரிவர்த்தனை அணுக எதிர்பார்க்கும் முகவரிகள் மற்றும் சேமிப்பக விசைகளைக் குறிப்பிடுகிறது, இது திறன் ஒப்பந்தங்களை உள்ளடக்கிய சிக்கலான பரிவர்த்தனைகளுக்கான [எரிவாயு](/developers/docs/gas/) செலவுகளைக் குறைக்க உதவுகிறது. EIP-1559 கட்டணச் சந்தை மாற்றங்கள் வகை 1 பரிவர்த்தனைகளில் சேர்க்கப்படவில்லை. வகை 1 பரிவர்த்தனைகளில் `yParity` அளவுருவும் அடங்கும், இது `0x0` அல்லது `0x1` ஆக இருக்கலாம், இது secp256k1 கையொப்பத்தின் y-மதிப்பின் சமநிலையைக் குறிக்கிறது. அவை `0x01` பைட்டுடன் தொடங்குவதன் மூலம் அடையாளம் காணப்படுகின்றன, மேலும் அவற்றின் TransactionType மதிப்பு `0x1` ஆகும்.

3. **வகை 2 பரிவர்த்தனைகள்**, பொதுவாக EIP-1559 பரிவர்த்தனைகள் என்று குறிப்பிடப்படுகின்றன, இவை எத்திரியத்தின் [லண்டன் மேம்பாட்டில்](/ethereum-forks/#london), [EIP-1559](https://eips.ethereum.org/EIPS/eip-1559)-இல் அறிமுகப்படுத்தப்பட்ட பரிவர்த்தனைகளாகும். அவை எத்திரியம் பிணையத்தில் நிலையான பரிவர்த்தனை வகையாக மாறிவிட்டன. இந்தப் பரிவர்த்தனைகள் பரிவர்த்தனை கட்டணத்தை அடிப்படை கட்டணம் மற்றும் முன்னுரிமைக் கட்டணம் எனப் பிரிப்பதன் மூலம் முன்கணிப்புத் திறனை மேம்படுத்தும் புதிய கட்டணச் சந்தை வழிமுறையை அறிமுகப்படுத்துகின்றன. அவை `0x02` பைட்டுடன் தொடங்குகின்றன மற்றும் `maxPriorityFeePerGas` மற்றும் `maxFeePerGas` போன்ற புலங்களை உள்ளடக்கியுள்ளன. வகை 2 பரிவர்த்தனைகள் அவற்றின் நெகிழ்வுத்தன்மை மற்றும் செயல்திறன் காரணமாக இப்போது இயல்புநிலையாக உள்ளன, குறிப்பாகப் பயனர்கள் பரிவர்த்தனை கட்டணங்களை மிகவும் கணிக்கக்கூடிய வகையில் நிர்வகிக்க உதவும் திறனுக்காக அதிகப் பிணைய நெரிசல் உள்ள காலங்களில் இவை விரும்பப்படுகின்றன. இந்தப் பரிவர்த்தனைகளுக்கான TransactionType மதிப்பு `0x2` ஆகும்.

4. **வகை 3 (தரவுத் திரளை) பரிவர்த்தனைகள்** எத்திரியத்தின் [டென்குன் மேம்பாட்டின்](/ethereum-forks/#dencun) ஒரு பகுதியாக [EIP-4844](https://eips.ethereum.org/EIPS/eip-4844)-இல் அறிமுகப்படுத்தப்பட்டன. இந்தப் பரிவர்த்தனைகள் "தரவுத் திரளை" (blob) தரவை (Binary Large Objects) மிகவும் திறமையாகக் கையாளும் வகையில் வடிவமைக்கப்பட்டுள்ளன, குறிப்பாக எத்திரியம் பிணையத்தில் குறைந்த செலவில் தரவை இடுகையிடுவதற்கான வழியை வழங்குவதன் மூலம் அடுக்கு 2 (l2) ரோலப்களுக்குப் பயனளிக்கின்றன. தரவுத் திரளைப் பரிவர்த்தனைகளில் `blobVersionedHashes`, `maxFeePerBlobGas` மற்றும் `blobGasPrice` போன்ற கூடுதல் புலங்கள் அடங்கும். அவை `0x03` பைட்டுடன் தொடங்குகின்றன, மேலும் அவற்றின் TransactionType மதிப்பு `0x3` ஆகும். தரவுத் திரளைப் பரிவர்த்தனைகள் எத்திரியத்தின் தரவு இருப்பு மற்றும் அளவிடுதல் திறன்களில் குறிப்பிடத்தக்க முன்னேற்றத்தைக் குறிக்கின்றன.

5. **வகை 4 பரிவர்த்தனைகள்** எத்திரியத்தின் [பெக்ட்ரா மேம்பாட்டின்](/roadmap/pectra/) ஒரு பகுதியாக [EIP-7702](https://eips.ethereum.org/EIPS/eip-7702)-இல் அறிமுகப்படுத்தப்பட்டன. இந்தப் பரிவர்த்தனைகள் கணக்குச் சுருக்கத்துடன் முன்னோக்கி இணக்கமாக இருக்கும் வகையில் வடிவமைக்கப்பட்டுள்ளன. அவை EOA-களை அவற்றின் அசல் செயல்பாட்டை சமரசம் செய்யாமல் தற்காலிகமாகத் திறன் ஒப்பந்தக் கணக்குகளைப் போலச் செயல்பட அனுமதிக்கின்றன. அவற்றில் `authorization_list` அளவுரு அடங்கும், இது EOA தனது அதிகாரத்தை எந்தத் திறன் ஒப்பந்தத்திற்கு வழங்குகிறது என்பதைக் குறிப்பிடுகிறது. பரிவர்த்தனைக்குப் பிறகு, EOA-இன் குறியீட்டுப் புலம் வழங்கப்பட்ட திறன் ஒப்பந்தத்தின் முகவரியைக் கொண்டிருக்கும்.
